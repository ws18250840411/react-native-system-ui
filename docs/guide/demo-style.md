# Demo 统一规范

> 目标：让每个组件的 demo 在“文案/结构/占位数据”上保持一致，方便横向对比视觉效果，减少与业务无关的干扰信息。

## 1. 标题与结构

- `docs/components/<component>.md` 的 “代码演示” 小标题与 `<code title="...">` 保持一致，标题命名尽量统一（如：基础用法 / 自定义 / 禁用状态 / 异步更新 / 徽标提示）。
- demo 代码尽量短：只展示组件本身，不额外堆叠说明性 `<Text>`；需要解释时放在 md 文档段落里。

## 2. 文案与占位数据

### 文案（通用）

- 纯占位展示：统一使用 `文字`。
- 选项类组件：优先使用 `选项一 / 选项二 / 选项三`（带描述时使用 `描述信息`）。
- 标签/导航类：优先使用 `标签名1 / 标签名2 / 标签名3`（或 `标签名`）。
- 复选框组：优先使用 `复选框a / 复选框b / 复选框c`。

### 数据（通用）

- 同一组件多个 demo 共享的 options，抽到 `docs/components/<component>/demo/options.ts` 复用，避免每个文件各写一套。
- 图片/媒体类 demo 统一使用稳定的占位资源（如 Vant 官方图片 URL），避免随意更换导致对比困难。

## 3. 状态展示与交互反馈

- **能不展示状态就不展示**：不在 demo 内写“当前选择：xxx / 当前状态：xxx”之类文案。
- 需要反馈交互时：
  - 优先用 `Toast.info(...)` 提示结果（更贴合移动端 demo 体验）。
  - 需要“表单式展示当前值”（如 DatetimePicker popup）时，优先用 `Field` 作为触发器与展示容器。

## 4. 代码风格（约定）

- 与仓库现有风格保持一致：单引号、无分号、简洁布局（常用 `View style={{ gap: 12 }}` 或 `Space`）。
- demo 内尽量避免复杂的自定义组件抽象；需要复用时优先抽到同目录下小文件（如 `options.ts`）。

## 5. 校验方式

- 每完成一个组件的 demo 调整：
  - 运行该组件相关单测（若存在）。
  - 运行 `pnpm docs:build` 确保文档站点可构建。
