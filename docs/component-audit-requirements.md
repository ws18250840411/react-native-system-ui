# 组件审查要求

> 本文档定义了组件库中每个组件需要达到的生产级别标准，作为全面梳理和审查的基准。

## 核心原则

### 1. 代码质量高、稳定性好
- ✅ **类型安全**：完整的 TypeScript 类型定义，避免使用 `any`（除非处理第三方库兼容性）
- ✅ **错误处理**：完善的错误边界处理，避免组件崩溃影响整体应用
- ✅ **边界情况**：处理所有可能的边界情况（null/undefined/空值/异常输入等）
- ✅ **内存管理**：避免内存泄漏，合理使用 ref 和清理函数
- ✅ **可访问性**：支持 ARIA 属性和无障碍访问

### 2. 精简易用
- ✅ **API 设计**：API 简洁直观，符合直觉
- ✅ **默认值**：合理的默认值，开箱即用
- ✅ **文档完整**：API 文档、示例代码、使用场景清晰
- ✅ **类型提示**：完善的 TypeScript 类型提示
- ✅ **避免过度设计**：组件设计简洁，不过度抽象，避免不必要的复杂性

### 3. 体积小
- ✅ **按需引入**：支持 Tree Shaking
- ✅ **依赖优化**：避免引入不必要的依赖
- ✅ **代码精简**：移除冗余代码，保持代码简洁
- ✅ **极致精简**：代码保持极致精简，清除不必要的注释，让代码自解释

### 4. 性能好
- ✅ **渲染优化**：合理使用 `useMemo`、`useCallback`、`React.memo`
- ✅ **依赖项优化**：`useMemo`/`useCallback` 的依赖项精确，避免不必要的重新计算
- ✅ **避免重复渲染**：减少不必要的组件重新渲染
- ✅ **缓存策略**：合理使用缓存，避免重复计算（注意内存管理）

### 5. 逻辑与官方一致
- ✅ **API 对齐**：与 React Vant 官方 API 保持一致
- ✅ **行为一致**：交互行为、状态管理逻辑与官方实现一致
- ✅ **实现细节**：参考官方实现细节，确保功能完整性

### 6. 文档保持一致
- ✅ **API 文档**：与 React Vant 官方文档保持一致
- ✅ **示例代码**：示例代码与官方 demo 对齐
- ✅ **使用场景**：文档说明与官方保持一致

### 7. 完全达到生产级别水平
- ✅ **测试覆盖**：完善的单元测试和集成测试
- ✅ **跨平台兼容**：iOS、Android、Web 三端一致性（详见"跨平台兼容检查"）
- ✅ **向后兼容**：API 变更保持向后兼容
- ✅ **性能监控**：关键路径性能可监控
- ✅ **可访问性**：支持 ARIA 属性和无障碍访问（参考 React Native Paper）
- ✅ **交互体验**：交互手感与系统原生组件一致（参考 React Native Paper）

## 审查清单

### 代码质量检查

- [ ] **类型安全**
  - [ ] 所有 props 都有完整的 TypeScript 类型定义
  - [ ] 避免使用 `any`（除非处理第三方库兼容性）
  - [ ] 事件回调参数类型正确（使用精确的事件类型，如 `GestureResponderEvent`、`NativeSyntheticEvent`）
  - [ ] Ref 类型定义正确（使用 `React.ElementRef` 或 `React.RefObject`）
  - [ ] 可推导的类型从 props 推导，避免冗余适配层

- [ ] **结构与渲染（跨端稳健）**
  - [ ] **避免 View 嵌套 Text**（导致 RN 崩溃）：所有"文本类 props"允许 `ReactNode` 时需保证内部渲染安全
  - [ ] `0/''/false` 等边界值渲染判定正确（避免被误判为"空"）
  - [ ] `style` 合并策略一致：容器/内容层级稳定，布局相关样式不丢失
  - [ ] 文本节点渲染：字符串/数字直接渲染，复杂节点使用 View 包裹
  - [ ] 条件渲染：使用 `null` 而非 `false` 或 `undefined` 避免警告

- [ ] **错误处理**
  - [ ] 自定义 render 函数（如 `iconRender`）有 try-catch 保护
  - [ ] 边界值检查完善（null/undefined/空值）
  - [ ] 错误状态有合理的降级处理
  - [ ] 错误信息清晰，便于调试

- [ ] **内存管理**
  - [ ] 使用 ref 存储数据时，有合理的清理机制
  - [ ] 缓存有大小限制，避免无限增长
  - [ ] `useEffect` 的清理函数正确实现
  - [ ] 避免闭包陷阱

- [ ] **可访问性（参考 React Native Paper）**
  - [ ] 支持 `accessibilityRole`、`accessibilityLabel`、`accessibilityState`
  - [ ] 支持键盘导航（Tab、Enter、Space、Arrow keys）
  - [ ] 支持屏幕阅读器（VoiceOver、TalkBack、NVDA）
  - [ ] 焦点管理：可聚焦元素有明确的焦点指示
  - [ ] 语义化：使用正确的 ARIA 角色和属性
  - [ ] 组件在三端都可被读屏理解（尤其是按钮、开关、输入、弹层）
  - [ ] 错误提示：表单错误有 `aria-describedby` 关联
  - [ ] 禁用状态：`accessibilityState={{ disabled: true }}` 正确设置

### 性能优化检查

- [ ] **渲染优化**
  - [ ] 合理使用 `useMemo` 缓存计算结果
  - [ ] 合理使用 `useCallback` 缓存回调函数
  - [ ] 必要时使用 `React.memo` 避免组件重新渲染
  - [ ] 依赖项精确，避免不必要的重新计算

- [ ] **计算优化**
  - [ ] 避免在渲染过程中进行复杂计算
  - [ ] 样式计算使用 `useMemo` 缓存
  - [ ] 事件处理函数使用 `useCallback` 缓存

- [ ] **缓存策略**
  - [ ] 缓存有合理的过期或清理机制
  - [ ] 缓存大小有限制，避免内存泄漏

### API 设计检查（参考 React Vant + React Native Paper）

- [ ] **API 简洁性（避免过度设计）**
  - [ ] **组件设计简洁**：不过度抽象，避免不必要的复杂性
  - [ ] **单一职责**：每个组件只做一件事，职责清晰
  - [ ] **避免过度封装**：不要为了"灵活性"而增加不必要的抽象层
  - [ ] **简单优先**：能用简单方案解决的，不要用复杂方案
  - [ ] **避免过度配置**：不要提供过多可配置项，保持 API 简洁
  - [ ] **代码可读性**：实现逻辑清晰，易于理解和维护

- [ ] **API 一致性（以 React Vant 为准）**
  - [ ] 与 React Vant 官方 API 保持一致
  - [ ] Props 命名与官方一致
  - [ ] 默认值与官方一致
  - [ ] 事件回调签名与官方一致
  - [ ] 受控/非受控行为一致（值来源、回调触发时机、边界值 clamp/normalize）

- [ ] **API 完整性**
  - [ ] 所有官方 API 都已实现
  - [ ] 受控/非受控模式都支持
  - [ ] Ref 方法完整
  - [ ] 事件语义一致：`onClick/onPress`、`onChange/onConfirm/onCancel` 等在三端触发条件一致

- [ ] **RN 端实现细节（参考 React Native Paper）**
  - [ ] 可达性与交互细节优先参考 RN Paper（更符合 RN 生态与系统行为）
  - [ ] 交互手感与系统原生组件一致
  - [ ] 使用 `@react-native-aria` 提供无障碍支持
  - [ ] 使用 `@react-stately` 管理状态（如适用）

- [ ] **平台差异处理**
  - [ ] 若 React Vant 与 RN Paper 冲突：
    - **对外 API 与语义** 优先跟随 React Vant（避免迁移成本与认知分裂）
    - **RN 端可达性与交互细节** 优先跟随 RN Paper（更符合 RN 生态与系统行为）
  - [ ] 仍无法一致时：允许"平台差异"，但必须 **可控（可配置/可禁用）** 且 **在组件文档明确说明**

### 文档检查

- [ ] **API 文档**
  - [ ] 所有 props 都有说明
  - [ ] 类型定义完整
  - [ ] 默认值说明清晰
  - [ ] 使用示例完整

- [ ] **示例代码**
  - [ ] 示例代码与官方 demo 对齐
  - [ ] 覆盖主要使用场景
  - [ ] 代码可运行

### 跨平台兼容检查（三端兼容性 - iOS/Android/Web）

- [ ] **平台检测与分支**
  - [ ] 使用 `Platform.OS === 'web'` 进行平台检测
  - [ ] Web 端特殊处理（如 DOM API、事件处理）有明确分支
  - [ ] 避免在 Native 端使用 Web 特有 API（如 `document`、`window`）
  - [ ] 避免在 Web 端使用 Native 特有 API（如原生模块）

- [ ] **动画兼容性**
  - [ ] **Web 下统一禁用 `useNativeDriver`**（避免 react-native-web 行为差异与警告）
  - [ ] 动画配置：`useNativeDriver: Platform.OS !== 'web'`
  - [ ] 动画性能：Native 端使用 native driver，Web 端使用 JS driver
  - [ ] 动画时长和缓动函数三端一致

- [ ] **样式兼容性**
  - [ ] **避免 View 嵌套 Text**（导致 RN 崩溃）：所有"文本类 props"允许 `ReactNode` 时需保证内部渲染安全
  - [ ] `0/''/false` 等边界值渲染判定正确（避免被误判为"空"）
  - [ ] `style` 合并策略一致：容器/内容层级稳定，布局相关样式不丢失
  - [ ] Web 端特殊样式（如 `position: 'fixed'`）使用条件渲染
  - [ ] Hairline 边框：Web 使用 `0.5px`，Native 使用 `StyleSheet.hairlineWidth`

- [ ] **事件处理兼容性**
  - [ ] Web 端触摸事件：使用 `dataSet: { noTouchSimulate: true }` 跳过 touch 模拟
  - [ ] 手势处理：Web 端支持鼠标拖拽和触摸，Native 端使用 PanResponder
  - [ ] 滚动冲突：Swipe/drag/scroll 冲突策略三端一致
  - [ ] 事件坐标：Web 端使用 `pageX/pageY`，Native 端使用 `nativeEvent`

- [ ] **弹层与定位（高风险区域）**
  - [ ] 所有弹层类组件（Popup/Overlay/Dialog/ActionSheet/Popover/DropdownMenu/Toast/ImagePreview…）必须走 **Portal**
  - [ ] zIndex 策略统一：OverlayStack 负责管理栈与"相对偏移"，避免小值污染全局层级
  - [ ] 定位必须覆盖非顶端场景：优先 `measureInWindow`；Web 需考虑布局/滚动导致的差异与兜底
  - [ ] Web 端 Portal 使用 `react-dom` 渲染，Native 端使用 PortalHost

- [ ] **输入与键盘（表单组件重点）**
  - [ ] TextInput：`onChangeText` / `onChange` / composition（中文输入法）/ focus/blur 时机一致
  - [ ] 键盘遮挡：Native 端使用 `KeyboardAvoidingView`，Web 端由业务层处理
  - [ ] 键盘避让策略明确；必要时提供"由业务层滚动容器处理"的建议与示例
  - [ ] disabled/readOnly 行为一致：是否可聚焦、是否触发回调、样式是否降级（需与 React Vant 语义对齐）

- [ ] **手势与滚动**
  - [ ] Swipe/drag/scroll 冲突策略一致（如 SwipeCell、Slider、Tabs、IndexBar）
  - [ ] Web 端需要明确：是否 `preventDefault`、是否支持鼠标拖拽、是否支持触摸
  - [ ] PanResponder 绑定到正确的容器，启用 move capture
  - [ ] 滚动性能：Web 端使用 `scrollEventThrottle`，Native 端使用原生滚动

- [ ] **文件与媒体处理**
  - [ ] Web 端文件选择：使用 `<input type="file">` 元素
  - [ ] Native 端文件选择：使用原生文件选择器或第三方库
  - [ ] 图片加载：Web 端支持 `src`，Native 端支持 `source`
  - [ ] 媒体资源清理：Web 端使用 `URL.revokeObjectURL` 清理 blob URL

- [ ] **安全区与布局**
  - [ ] 安全区适配：使用 `SafeAreaView` 或 `useSafeAreaInsets`
  - [ ] 底部安全区：Native 端自动适配，Web 端可配置
  - [ ] 状态栏高度：Native 端使用 `StatusBar.currentHeight`，Web 端为 0

- [ ] **性能优化（跨平台）**
  - [ ] 减少布局测量次数：避免在 render 中触发测量
  - [ ] 关键路径避免不必要的 state 抖动
  - [ ] Web 端避免频繁的 DOM 操作
  - [ ] Native 端合理使用原生驱动动画

### 体积与依赖检查（参考 React Vant + React Native Paper）

- [ ] **依赖优化**
  - [ ] 不新增重依赖；优先复用现有 hooks/utils
  - [ ] 避免引入不必要的第三方库
  - [ ] 使用 peerDependencies 而非 dependencies（如适用）

- [ ] **代码精简（避免过度设计）**

  - [ ] **避免过度分层**：不要为了"架构"而增加不必要的中间层
  - [ ] **直接实现优先**：能直接实现的，不要过度设计模式

## 审查流程

1. **设计审查（避免过度设计）**
   - 检查组件设计是否简洁，是否过度抽象

   - 检查代码质量、性能优化、错误处理
   - 检查代码是否简洁，避免过度设计
   - 对照官方实现，确保逻辑一致

3. **文档审查**
   - 检查 API 文档完整性
   - 检查示例代码与官方对齐
   - 检查文档是否清晰简洁

4. **测试审查**
   - 检查测试覆盖率
   - 运行测试确保通过

5. **跨平台测试**
   - 在 iOS、Android、Web 三端测试
   - 确保行为一致性

6. **性能测试**
   - 检查关键路径性能
   - 确保无内存泄漏

## 设计原则总结

### 核心设计理念

1. **简洁优先**：能用简单方案解决的，不要用复杂方案
2. **避免过度设计**：组件设计简洁，不过度抽象，避免不必要的复杂性
3. **单一职责**：每个组件只做一件事，职责清晰
4. **直接实现**：能直接实现的，不要过度设计模式
5. **可维护性**：代码简洁清晰，易于理解和维护

### 过度设计的常见表现（需避免）

- ❌ 为了"灵活性"而增加不必要的抽象层
- ❌ 提供过多可配置项，导致 API 复杂
- ❌ 过度使用设计模式，增加理解成本
- ❌ 不必要的中间层和适配层
- ❌ 单个组件文件过大（超过 500 行）
- ❌ 状态管理过度复杂（能用 `useState` 解决的用复杂方案）

### 简洁设计的标准

- ✅ 组件职责单一，功能清晰
- ✅ API 简洁直观，符合直觉
- ✅ 代码逻辑清晰，易于理解
- ✅ 实现方案简单直接
- ✅ 维护成本低

## 常见问题与解决方案

### 1. 如何判断是否过度设计？
- **检查点**：组件职责是否单一？API 是否简洁？代码是否可以直接实现？
- **解决方案**：对照官方实现，如果官方用简单方案，我们也用简单方案

### 2. 如何处理 React Vant 与 RN Paper 的冲突？
- **原则**：对外 API 优先 React Vant，RN 端实现细节优先 RN Paper
- **解决方案**：在文档中明确说明平台差异，提供配置选项

### 3. 如何确保三端兼容性？
- **检查点**：使用 `Platform.OS` 进行平台检测，Web 下禁用 `useNativeDriver`
- **解决方案**：在 iOS、Android、Web 三端分别测试，确保行为一致

### 4. 如何避免内存泄漏？
- **检查点**：ref 存储的数据是否有清理机制？缓存是否有大小限制？
